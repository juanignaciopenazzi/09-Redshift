¿Dónde encaja Redshift?

  • S3 es el storage del Data Lake (raw / curated / analytics).
  • Glue cataloga metadatos y corre ETL.
  • RDS / Aurora resuelve workloads transaccionales (OLTP).
  • Redshift es el Data Warehouse: datos estructurados listos para reporting, dashboards y SQL analítico (OLAP).
  • No reemplaza al Data Lake ni a RDS: complementa el stack hacia analytics.
teraclud

OLTP vs OLAP

OLTP — RDS / Aurora

  • Muchas escrituras chicas: INSERT, UPDATE, DELETE.
  • Operaciones sobre pocas filas (un pedido, un usuario).
  • Baja latencia para la aplicación.
  • Integrity y concurrency de transacciones.

OLAP — Redshift

  • Lecturas sobre volúmenes grandes.
  • Agregaciones, joins, trends históricos.
  • Workloads de BI y analytics.
  • Queries que escanean y resumen datasets.

Correr dashboards pesados contra RDS productivo impacta a la app y al costo.

Qué es un Data Warehouse

  • Repositorio central de datos estructurados, modelados para análisis — no para la app transaccional.
  • Schema-on-write: definís tablas antes de cargar; los datos tienen que cumplir el schema.
  • Orientado a historial, métricas y preguntas de negocio.
  • Convive con el Data Lake: el lake es flexible y de bajo costo de storage; el Data Warehouse optimiza performance y consistencia para BI.
  • Patrón habitual: cargar desde S3 (capa curated) hacia Redshift.
teraclud

Amazon Redshift

  • Data Warehouse managed de AWS, con SQL compatible con PostgreSQL.
  • Storage columnar: lee solo las columnas que necesita la query.
  • Escala a datasets grandes y queries analíticas concurrentes.
  • Se integra con S3 (COPY / UNLOAD), Glue Data Catalog, Spectrum, Data Sharing, QuickSight, etc.
  • Hoy trabajamos con Redshift Serverless: sin administrar cluster, billing por compute usado.
teraclud

Provisioned vs Serverless

Provisioned (cluster)

  • Elegís node type y cantidad de nodos.
  • Corre de forma continua mientras el cluster exista.
  • Más control operativo y de sizing.
  • Útil con workload predecible y sostenido.

Serverless

  • Se arma con Namespace + Workgroup.
  • Capacidad medida en RPU; scale automático.
  • Cuando no hay queries, el compute se detiene.
  • Útil con uso intermitente o variable.

Serverless no es gratis: se factura compute cuando hay actividad, más storage. Al cerrar, eliminar Workgroup y Namespace.

Namespace vs Workgroup

Namespace

  • Contenedor de datos y metadata.
  • Databases, schemas, tables, users.
  • Admin credentials, IAM roles, encryption (KMS).
  • Es “qué datos tengo y quién puede administrarlos”.

Workgroup

  • Capa de compute y red.
  • Base RPU, VPC, subnets, security groups.
  • Endpoint al que te conectás a queryar.
  • Es “con cuánta capacidad y desde dónde corro SQL”.

Un Namespace se asocia a un Workgroup. Los datos viven en el Namespace; el Workgroup aporta el compute.

teraclud

RPU: capacidad y billing

  • RPU = Redshift Processing Unit. Unidad de compute de Serverless (CPU, memoria y red asociadas).
  • Base capacity: RPU mínimos configurados en el Workgroup. Más RPU → más capacidad de compute (queries más grandes o más concurrentes).
  • Billing de compute: se cobra por RPU-hour, en segundos de actividad. Si no hay workload, no facturás compute.
  • Storage: se cobra por separado (datos persistidos en el Namespace), típicamente por GB-mes.
  • Para este workshop: base RPU baja y datasets acotados. Borrar Workgroup + Namespace al cerrar.
teraclud

Por qué el almacenamiento columnar importa en analytics

  • OLTP suele necesitar la fila completa; analytics suele necesitar pocas columnas + agregaciones.
  • Columnar reduce I/O, mejora compresión y acelera SUM, AVG, GROUP BY.
  • Misma idea que Parquet en el Data Lake, aplicada dentro del Data Warehouse.
  • SELECT barrio, SUM(monto) no tiene que leer columnas irrelevantes del row.

Carga de datos: COPY desde S3

  • Patrón estándar: S3 → Redshift con COPY (bulk load paralelo).
  • INSERT fila por fila no escala para cargas analíticas.
  • Requisitos: tabla destino, path S3, IAM role con acceso al bucket, formato (CSV, Parquet, etc.).
  • UNLOAD es el camino inverso: resultados de Redshift hacia S3.
  • Es el bridge natural con el Data Lake de las clases anteriores.

Redshift vs Athena

Athena

  • Query sobre el Data Lake sin mover datos.
  • Schema-on-read + Glue Data Catalog.
  • Billing por data scanned.
  • Fuerte para exploración y validación.

Redshift

  • Datos cargados al Data Warehouse (o vía Spectrum).
  • Schema-on-write, optimizado para BI.
  • Serverless: billing por RPU-hour + storage.
  • Fuerte para dashboards recurrentes y SQL complejo.

No son excluyentes: Athena para explorar el lake; Redshift para reporting estable y concurrency controlada.

Data Sharing

  • Permite compartir datos en vivo entre namespaces/clusters de Redshift — sin copiar datasets con ETL.
  • Producer otorga acceso (datashare) a schemas, tables o views; consumer los consulta como objetos locales.
  • Sirve para multi-account, multi-team y data mesh: una fuente de verdad, varios consumidores.
  • Reduce duplicación, costo de storage y riesgo de versiones desfasadas.
  • Concepto clave en entornos productivos; en el workshop lo documentamos a nivel de diseño (sin implementar cross-account obligatorio).

Redshift ML y LLMs con Bedrock

  • Redshift ML: crear y usar modelos de machine learning desde SQL (CREATE MODEL), sin sacar los datos del Data Warehouse. Por debajo se apoya en SageMaker para training/inference clásico.
  • Integración con Amazon Bedrock: con CREATE EXTERNAL MODEL apuntás a un foundation model (Claude, Titan, Llama, Mistral, etc.) y Redshift genera una función SQL de inference.
  • Invocás el LLM con un SELECT sobre tus tablas: summarization, sentiment analysis, clasificación, traducción, generación de texto.
  • El valor: analytics + GenAI en el mismo lugar — sin exportar datasets a otro servicio solo para enriquecer texto.
  • Hay costo de Bedrock (tokens del modelo) además del compute de Redshift. Vale la pena cuando el dato ya vive en el warehouse.
teraclud

SORTKEY y DISTSTYLE

  • SORTKEY: orden físico de los datos. Filtrar por esa columna (ej. fecha) reduce I/O.
  • DISTSTYLE / DISTKEY: cómo se distribuyen las filas entre nodos. Impacta JOINs grandes.
  • Para este ejercicio: DISTSTYLE AUTO y un SORTKEY alineado al filtro más frecuente.
  • Optimizar dist keys sin entender el workload suele ser prematuro; primero modelo correcto y queries claras.

Buenas prácticas

  • Cargar con COPY, no con inserts masivos.
  • Preferir Parquet (o CSV limpio) desde curated cuando esté disponible.
  • Configurar base RPU conservadora y monitorear uso.
  • Tags en Namespace/Workgroup: owner, env, workshop.
  • Entender Data Sharing antes de duplicar datasets entre equipos.
  • Documentar en inglés y eliminar Workgroup + Namespace al terminar.
